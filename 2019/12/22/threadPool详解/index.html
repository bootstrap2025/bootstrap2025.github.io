    关于线程池有这样一个介绍：

    为什么需要创建线程池

    • 降低资源消耗，通过重复利用已创建的线程降低线程创建和销毁造成的消耗。
    • 提高响应速度，当任务到达时，任务可以不需要通过等到线程创建就可以立即执行。
    • 提高线程池的管理性，线程是稀缺资源，如果无限制创建线程，不仅会消耗系统资源，还会降低系统的稳定性，有说单独启动一个线程占用的内存是1M左右，使用线程池可以对线程统一分配、调优和控制。

    线程池的创建方式

    • newFixedThreadPool适用于负载较重的服务，固定线程数，允许请求的最大队列数Integer.MAX_VALUE1；
    • SingleThreadExecutor 也就是单线程池，单个worker线程的exceutor，适用于单个线程的服务，允许请求的最大队列数Integer.MAX_VALUE；
    • CachedThreadPool 根据需要创建新线程的线程池，适用多个轻量级线程的服务，允许创建的最大线程数是Integer.MAX_VALUE；
    • newWorkStealingPool jdk8添加的新的线程池，内部构建了ForkJoinPool；
    • ScheduledThreadPoolExecutor 可以进行定时或者周期性的工作调度；
    • 可以用spring里的ThreadPoolTaskExecutor 创建web应用线程，底层也是封装的ThreadPoolExecutor；
      另外，1-5 是Executors 提供的5个可以创建的线程池，其中1-3底层调用的都是ThreadPoolExecutor的构造方法；

    ThreadPoolExecutor线程池的执行流程

    线程池的执行步骤
    • 提交任务；
    • 判断核心线程池是否已满，如果已满执行步骤3，否则，创建新的线程；
    • 判断是否队列已满，如果队列未满存储在阻塞队列，否则执行步骤4；
    • 判断最大线程池是否已满，如果满了执行拒绝策略，否则创建线程任；

    之所以采用如上步骤设计线程池，是为了在执行exectue()方法时，尽可能的避免全局锁，在ThreadPoolExecutor完成预热后（当前运行的线程数大于等于CorePoolSize）时，几乎所有所有的execute()方法都执行步骤2，而步骤二不需要获取全局锁。

    ctl变量

    阅读ThradPoolExecutor源码时经常会看到ctl变量，在源码里可以看到

    ctl变量是AtomicInteger的变量，该变量将线程状态值和最大工作线程数巧妙的放在一个属性里，对于Integer共32位，前3位代表状态值，后29位代表线程池的线程数。

    线程池的五种状态
    • RUNNING 运行中，可处理新任务并执行队列中的任务，线程池一旦被创建，就处于RUNNING状态,线程池任务数为0；
    • SHUTDOWN 关闭，不接受新任务，但处理队列中的任务,执行shutdown之后的状态；
    • STOP 停止，不接受新任务，不处理队列中任务，且打断运行中任务,执行shutdownNow之后的状态；
    • TIDYING 整理所有任务已经结束，workerCount = 0，将执行terminated()方法，线程数为0，过渡到terminated(）方法；
    • TERMINATED 结束 terminated()方法结束后的线程池状态，线程池的彻底终止状态；

    线程池的创建步骤：

    根据任务选择合适的线程池创建方式
    • Executors 是工厂类，采用ThreadPoolExecutor手动生成（推荐）；
      参数:int corePoolSize 核心线程数
      int maximumPoolSize 最大线程数
      long keepAliveTime ：当线程池中的空闲线程数超过corePoolSize时，多余线程多久会被销毁
      TimeUnit unit
      BlockingQueue workQueue
      ThreadFactory threadFactory
      RejectedExecutionHandler handler
    • 采用spring ThreadPoolTaskExecutor 创建线程池；
    选择线程池的拒绝策略：
    • AbortPolicy(默认)：丢弃任务并抛出RejectedExecutionException异常；
    • DiscardPolicy：正常执行不处理；
    • DiscardOldestPolicy： 抛弃队列中等待最久的任务，把当前队列任务加入队列中再次提交当前线程；
    • CallerRunsPolicy： 策略既不会抛弃任务，也不会抛出异常，而是将任务回馈至发起方比如main线程；
    线程池任务的提交：
    • execute(Runnable task)方法是没有返回值，提交的是Runnable对象；
    • submit(Callable task)方法是提交submit方法，提交的Callabe或者Runable对象，submit底层实现也是采用execute(Runnable command)；
